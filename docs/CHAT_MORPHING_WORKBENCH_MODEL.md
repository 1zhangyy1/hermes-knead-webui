# Chat 变形工作台模型

日期：2026-05-31

本文用于梳理当前最新产品理解：Next AI 不是工作台管理器，也不是 Lovable 式应用生成器，而是一个会根据任务和用户习惯逐步变形的 Chat。

## 1. 最核心的一句话

> 普通 Chat 是初始形态；工作台是 Chat 在某类任务中长出来、并被保存下来的任务形态；自进化是 Chat 在用户需要时，把自己的流程、默认行为和页面形态改得更适合这件事。

更口语化：

> 用户不是来造应用的，用户是在做事。AI 为了让这件事做得更顺，会把当前 Chat 页面变得越来越像这类任务需要的产品。

## 2. 和 Lovable 的区别

Lovable 的心智：

```text
我描述一个应用
AI 帮我生成一个应用
我预览、修改、发布这个应用
```

Next AI 的心智：

```text
我在 Chat 里做事
Chat 根据任务阶段长出合适的界面
我越用，它越懂我的工作方式
以后同类任务自动进入更合适的形态
```

关键区别：

- Lovable 的对象是“要被构建的应用”。
- Next AI 的对象是“用户当前正在做的任务”。
- Lovable 的 AI 是 builder。
- Next AI 的 AI 首先是 worker，必要时才变成 page improver。

## 3. 三层修改关系

用户在 Chat 里说的话，AI 要判断它属于哪一层。

### 3.1 改当前内容

只影响当前任务结果。

例子：

```text
把第 3 页写短一点
这个标题更有冲击力
给这一页加讲稿
```

AI 行为：

```text
直接修改当前 PPT / 文案 / 讲稿
不改变以后默认
不改变页面形态
```

### 3.2 改工作方式

影响以后同类任务的默认行为。

例子：

```text
以后每页都带讲稿
以后融资 PPT 文案短一点
每次开始前先问我页数和受众
```

AI 行为：

```text
先完成当前任务
再询问是否记住为默认习惯
用户确认后写入工作方式
可撤销
```

用户感知：

```text
已记住：PPT 默认短文案 + 每页讲稿
下次自动使用
[撤销]
```

### 3.3 改页面形态

影响 Chat 页面或当前工作台的布局、模块和任务界面。

例子：

```text
把讲稿区放右边
我想边看页面边跟你聊
以后做 PPT 时一直显示缩略图
这个界面太挤了，把大纲收起来
```

AI 行为：

```text
识别这是页面形态修改
生成轻量改造建议
先预览
用户确认后应用到当前工作台
可撤销 / 可恢复
```

用户感知：

```text
我可以把 PPT 工作台调整成：
- 中间显示当前页面
- 底部显示缩略图
- 右侧固定讲稿区

[预览布局] [以后默认] [只这次]
```

## 4. AI 和页面的关系

用户表面上只面对一个 AI，但这个 AI 在不同语境下有不同职责。

```text
同一个 AI
├─ 做任务：改 PPT、写文案、生成页面
├─ 记习惯：保存用户偏好和默认流程
└─ 改页面：在用户需要时调整 Chat/工作台形态
```

重要的是：不要把这些职责做成三个显式角色名让用户理解。

用户只需要知道：

> AI 可以帮我做事，也可以在我需要时把这个页面改得更适合这件事。

## 5. 触发机制

页面变化有两种触发。

### 5.1 用户明确要求

用户直接说出页面或默认流程诉求：

```text
以后都这样
默认这样
把这个区域放右边
我想一直看到讲稿
下次不用再问这个
```

AI 可以立即提出改造建议。

### 5.2 系统主动发现

系统观察到重复模式：

```text
用户每次都打开讲稿区
用户每次都要求短文案
用户每次都把缩略图放到底部
用户多次跳回第 3 页修改
用户多次选择同一种风格
```

AI 可以克制地提出：

```text
我注意到你经常打开讲稿区。
要不要以后默认显示讲稿？

[以后默认显示] [不用]
```

主动建议原则：

- 不频繁打扰。
- 有明确收益才提。
- 一次只提一个小改动。
- 用户必须确认。
- 随时可撤销。

## 6. AI PPT 任务里的页面变形

AI PPT 是最适合做第一条完整 demo 的场景。

### 阶段 1：普通 Chat

```text
用户：做一个介绍 Codex 的 PPT
```

页面：

```text
普通 Chat
```

### 阶段 2：Chat 里长出信息卡

页面：

```text
Chat
  关键信息卡：
    主题
    场景
    页数
    比例
    语言
    风格
```

AI 行为：

```text
先收集足够信息，不直接乱生成。
```

### 阶段 3：进入 PPT 生成模式

页面：

```text
左侧：AI 对话和任务步骤
右侧：关键信息、风格选择、生成按钮
```

Chat 仍然在，只是页面开始围绕 PPT 任务变形。

### 阶段 4：生成后变成 PPT 编辑器

页面：

```text
左侧：AI 对话 / 操作历史 / 当前步骤
中间：PPT 页面画布
底部：页面缩略图
右侧：大纲 / 讲稿 / 页面属性
```

用户仍然可以用自然语言控制：

```text
第 3 页重做，更像发布会
把这一页讲稿压短
给第 5 页加一张对比图
```

### 阶段 5：保存工作方式

当用户说：

```text
以后做技术介绍 PPT 都这样
```

或者系统观察到重复偏好后，AI 提示：

```text
要不要把这套方式设为技术介绍 PPT 的默认方式？

包括：
- 先收集主题、受众、页数
- 默认生成讲稿
- 文案偏短
- 生成后进入页面编辑模式

[以后默认这样] [只这次]
```

下次用户进入 PPT 工作台时，不再从纯空白 Chat 开始，而是直接进入更适合 PPT 的起始形态。

## 7. 工作台到底是什么

工作台不是一个固定空间，也不是一个独立 SaaS 页面。

工作台是：

```text
Chat 的某种稳定任务形态
```

例如：

```text
PPT 工作台 = Chat 的 PPT 形态
研究工作台 = Chat 的研究形态
数据工作台 = Chat 的数据分析形态
```

它保存的是：

- 默认任务类型。
- 默认提问流程。
- 用户偏好。
- 页面形态。
- 可用模块。
- 输出结构。
- 历史任务和产物。

## 8. 技术实现方向

第一版不要让 AI 直接任意改代码。先做状态驱动的页面变形。

### 8.1 任务状态

每个会话背后有一个 `task_state`：

```json
{
  "type": "ppt",
  "phase": "collecting_info",
  "subject": "Codex 介绍",
  "scenario": "技术产品介绍",
  "pageCount": "auto",
  "ratio": "16:9",
  "language": "简体中文",
  "style": null,
  "slides": [],
  "selectedSlide": null
}
```

### 8.2 页面形态渲染器

UI 根据 `task_state` 渲染不同 surface：

```text
none
  -> 普通 Chat

ppt + collecting_info
  -> Chat + 信息卡

ppt + choosing_style
  -> Chat + 风格选择

ppt + generating
  -> Chat + 生成进度

ppt + editing
  -> Chat + 画布 + 缩略图 + 讲稿
```

### 8.3 工作方式默认值

工作台保存默认值：

```json
{
  "id": "ppt-workbench",
  "taskType": "ppt",
  "defaults": {
    "askFirst": ["主题", "受众", "页数"],
    "includeSpeakerNotes": true,
    "copyStyle": "短文案",
    "layout": "canvas-with-thumbnails-and-notes"
  }
}
```

### 8.4 AI 可调用的最小工具

```text
create_task(type, initial_state)
update_task_state(patch)
set_surface(surface_id)
remember_preference(key, value)
suggest_surface_change(change)
apply_surface_change(change_id)
undo_last_preference()
```

AI 通过这些工具改变任务状态和页面形态，而不是直接碰核心代码。

## 9. 第一版 MVP

只做 AI PPT 一条线。

必须跑通：

```text
1. 用户从普通 Chat 发起 PPT 任务
2. Chat 出现关键信息卡
3. 用户确认后进入 PPT 生成模式
4. 生成后进入 PPT 编辑模式
5. 用户通过 Chat 修改页面
6. 用户说“以后都这样”
7. 系统保存默认工作方式
8. 下次 PPT 任务自动进入更贴合的形态
9. 用户可以撤销这个默认设置
```

暂时不做：

- 任意应用生成。
- 复杂工作台市场。
- 多种工作台同时铺开。
- AI 直接改 Host Core。
- 复杂版本 diff。
- 让用户理解 manifest / generated app。

## 10. 产品原则

1. Chat-first，但不是 Chat-only。
2. 页面可以变，但必须因任务而变。
3. 默认不乱变。
4. 用户明确要求时，可以改页面。
5. 系统主动建议必须克制。
6. 每次改变都要能解释、能预览、能撤销。
7. 改变只影响当前工作台，不影响母体应用。
8. 用户感知应该是“越来越好用”，不是“我在造应用”。

## 11. 当前最清晰的表达

> Next AI 是一个会变形的 Chat。普通 Chat 是起点；当用户做 PPT、研究、数据分析时，Chat 会按任务阶段长出更合适的界面。用户可以要求它记住习惯、调整流程、改变页面形态。它不会乱变，每次改变都能预览、确认和撤销。工作台就是这些被保存下来的任务形态。
