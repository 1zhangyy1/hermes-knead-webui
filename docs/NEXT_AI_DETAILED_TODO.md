# Next AI 详细 TODO：从 Agent Chat 到自进化工作台

日期：2026-05-29

本文是当前执行清单。产品定义以 `docs/PRODUCT_DIRECTION_CURRENT.md` 为准；本文只回答：接下来具体怎么做、先做什么、做到什么程度算完成。

## 0. 当前总判断

我们现在不要先做一个复杂的“工作台平台”，也不要一上来让 AI 直接改母体应用。

第一版主线是：

```text
极简 Agent Chat
  -> 任务中长出临时任务界面
  -> 保存为专属工作台
  -> 工作台可预览 / 应用 / 回滚地自进化
```

参考 PilotDeck 的地方：

- WorkSpace 是操作边界。
- 左侧管理工作区和会话。
- 中央永远保留 Agent Chat。
- 工具调用、思考过程、执行过程可见。
- 文件、记忆、技能要按工作区沉淀。

我们比 PilotDeck 多出来的核心：

- 工作台不只是隔离文件和记忆，还要沉淀 UI 和流程。
- 专属界面不是用户先选出来的，而是从任务中长出来的。
- 自进化必须有预览、应用、回滚。

## 1. MVP 0：把基础 Agent Chat 做清楚

目标：用户打开产品时，一眼知道“这是一个可以开始工作的 AI Agent”。

### 1.1 界面收敛

- [ ] 左侧只保留必要入口：新对话、会话、工作台、设置。
- [ ] 默认入口必须是“新对话”，不是“新建工作台”。
- [ ] 中央默认是 Chat 流和输入框。
- [ ] 隐藏第一版不必要的复杂概念：连接器、技能市场、自动化、复杂模型配置。
- [ ] 保留思考过程和工具调用卡片，不把 Agent 过程藏掉。
- [ ] 输入框只保留高频操作：附件、输入、发送。

完成标准：

- [ ] 新用户不需要理解 Space / manifest / generated app，也能开始聊天。
- [ ] 当前页面看起来像一个干净的 Agent，而不是管理后台。
- [ ] Hermes Agent 能稳定收到消息并返回结果。

### 1.2 Agent 运行可用

- [ ] 确认 OpenRouter 配置可用。
- [ ] 默认模型先使用 `anthropic/claude-sonnet-4.6`。
- [ ] 聊天失败时显示明确错误，不静默失败。
- [ ] 会话刷新后仍可恢复。
- [ ] 工具调用状态能在聊天流中显示。

完成标准：

- [ ] 用户输入普通问题，可以得到回答。
- [ ] 用户要求读写文件时，Agent 有明确权限和过程反馈。
- [ ] 断线或失败后用户知道发生了什么。

## 2. MVP 1：任务界面在 Chat 中长出来

目标：用户仍然在和 Agent 聊天，但某些任务会自然出现结构化任务面板。

第一版只做三个任务类型的最小面板：

- PPT 任务：大纲 / 页面 / 讲稿。
- 研究任务：资料 / 卡片 / 草稿。
- 数据任务：表格 / 图表 / 报告。

### 2.1 任务识别

- [ ] 在前端先做轻量规则识别，不等完整模型能力。
- [ ] 识别 PPT 关键词：PPT、slide、deck、路演、融资、演示文稿。
- [ ] 识别研究关键词：调研、研究、资料、竞品、论文、报告。
- [ ] 识别数据关键词：表格、Excel、csv、数据、图表、分析。
- [ ] 识别结果只触发 UI 面板，不改变模型回答逻辑。

完成标准：

- [ ] 用户说“帮我做一个融资 PPT”，页面出现 PPT 任务面板。
- [ ] 用户说“调研一下某个市场”，页面出现研究任务面板。
- [ ] 用户说“分析这个表格”，页面出现数据任务面板。

### 2.2 任务面板结构

- [ ] 新增一个轻量任务状态层：当前任务类型、当前阶段、当前对象。
- [ ] 任务面板不抢走 Chat，Chat 始终在场。
- [ ] PPT 面板包含：大纲列表、页面列表、讲稿区。
- [ ] 研究面板包含：资料卡片、要点、草稿。
- [ ] 数据面板包含：数据预览、分析结论、图表占位。
- [ ] 面板内容第一版可以前端模拟，但要和聊天任务相关。

完成标准：

- [ ] 用户能在 Chat 里指挥“改第 3 页”。
- [ ] UI 能表达当前操作对象是第 3 页。
- [ ] 面板不是静态宣传卡，而是任务过程的一部分。

## 3. MVP 2：保存为工作台

目标：任务做完后，用户可以把这套流程保存成可复用的工作台。

### 3.1 保存时机

- [ ] 不在首页强推工作台。
- [ ] 只在用户完成一次明确任务后提示保存。
- [ ] 提示语要自然：这类任务以后要不要保存成工作台？
- [ ] 用户可以选择：保存、预览、暂时不要。

完成标准：

- [ ] 用户做完 PPT 任务后，出现“保存为 PPT 工作台”的建议。
- [ ] 用户不保存时，产品不会继续打扰。

### 3.2 工作台数据结构

- [ ] 定义 `workbench.json` 或统一状态文件。
- [ ] 每个工作台至少保存：
  - id
  - name
  - type
  - description
  - source session
  - system prompt
  - task schema
  - UI layout
  - artifacts
  - version
  - created at / updated at
- [ ] 用户语言里不暴露这些内部字段。

建议结构：

```json
{
  "id": "ppt-workbench",
  "name": "PPT 工作台",
  "type": "ppt",
  "description": "用于把材料整理成大纲、页面和讲稿。",
  "version": 1,
  "sourceSessionId": "session-id",
  "agentPrompt": "你是一个帮助用户完成 PPT 的工作台 Agent。",
  "taskSchema": {
    "inputs": ["主题", "受众", "目标", "材料"],
    "outputs": ["大纲", "页面", "讲稿"]
  },
  "uiLayout": {
    "panels": ["chat", "outline", "slides", "notes"]
  },
  "status": "enabled"
}
```

完成标准：

- [ ] 保存后左侧工作台列表出现新工作台。
- [ ] 刷新页面后工作台仍存在。
- [ ] 点击工作台能回到对应的 Chat + 任务面板形态。

## 4. MVP 3：工作台可自进化

目标：工作台可以根据用户反馈改自己的流程和界面，但第一版先改配置，不直接改母体代码。

### 4.1 改造入口

- [ ] 用户可以在 Chat 中说：以后这个工作台先问我页数。
- [ ] 用户可以说：把大纲区放左边。
- [ ] 用户可以说：默认输出三种风格。
- [ ] 系统生成一个“改造建议卡”，而不是直接应用。

改造建议卡必须包含：

- [ ] 改什么。
- [ ] 为什么改。
- [ ] 改完大概长什么样。
- [ ] 影响哪些内容：Prompt、流程、UI、默认字段。
- [ ] 操作：预览、应用、取消。

完成标准：

- [ ] 用户能从 Chat 触发工作台改造建议。
- [ ] 改造建议不直接生效。

### 4.2 预览 / 应用 / 回滚

- [ ] 每次改造生成新版本。
- [ ] 新版本可以预览。
- [ ] 应用后 current version 指向新版本。
- [ ] 回滚后恢复上一个版本。
- [ ] 版本历史可查看，但第一版不用做复杂 diff。

完成标准：

- [ ] v1 和 v2 可以并存。
- [ ] 应用 v2 后刷新仍是 v2。
- [ ] 回滚后回到 v1。

### 4.3 第一版允许改什么

第一版只允许改：

- [ ] 工作台 system prompt。
- [ ] 默认任务流程。
- [ ] 表单字段。
- [ ] 面板显示顺序。
- [ ] 面板开关。
- [ ] 输出模板。

第一版暂不允许改：

- [ ] Host Core。
- [ ] 路由系统。
- [ ] 权限系统。
- [ ] 任意 npm 依赖。
- [ ] 全局 UI。

## 5. MVP 4：生成式专属子应用

目标：当配置化工作台不够用时，允许 AI 生成独立子应用代码，但必须隔离。

### 5.1 目录边界

- [ ] 新增 `spaces/generated/` 或 `workbenches/generated/`。
- [ ] 每个工作台一个目录。
- [ ] 每个版本一个目录。
- [ ] Host 只读 manifest，不直接依赖内部实现。

建议目录：

```text
spaces/generated/
  ppt-workbench/
    manifest.json
    versions/
      v1/
        index.html
        app.js
        style.css
      v2/
        index.html
        app.js
        style.css
    current
```

完成标准：

- [ ] 生成代码和 Host Core 分离。
- [ ] 子应用可以预览。
- [ ] 子应用坏了不影响主 Chat。

### 5.2 安全机制

- [ ] 应用前必须保存快照。
- [ ] 应用前必须跑最小检查。
- [ ] 失败不能应用。
- [ ] 用户能回滚。
- [ ] 第一版禁止子应用任意访问用户文件。

完成标准：

- [ ] 生成错误不会拖垮主应用。
- [ ] 用户永远能回到上一个可用版本。

## 6. UI 信息结构 TODO

目标：第一版页面足够简单，用户不被概念淹没。

### 6.1 左侧栏

- [ ] 顶部：产品名。
- [ ] 主操作：新对话。
- [ ] 会话列表：最近对话。
- [ ] 工作台列表：用户保存过的工作台。
- [ ] 底部：设置。
- [ ] 不展示：连接器、技能、自动化、市场、复杂项目分组。

### 6.2 中央区域

- [ ] 默认空态：一句清晰问题，比如“今天想让 AI 帮你做什么？”
- [ ] Chat 流：用户消息、AI 回复、工具过程。
- [ ] 输入框：简洁、稳定、不要塞满配置。
- [ ] 任务面板可以在右侧或下方出现，但不能替代 Chat。

### 6.3 工作台视图

- [ ] 打开工作台后仍然是 Chat。
- [ ] 工作台只改变上下文、任务面板和默认流程。
- [ ] 顶部可以显示当前工作台名。
- [ ] 工作台内部可以有材料、结果、版本、改造建议。

## 7. 后端 / 数据 TODO

### 7.1 工作台 API

- [ ] `GET /api/workbenches`：列出工作台。
- [ ] `POST /api/workbenches/save`：从当前任务保存工作台。
- [ ] `GET /api/workbenches/:id`：读取工作台详情。
- [ ] `POST /api/workbenches/:id/evolve`：生成改造建议。
- [ ] `POST /api/workbenches/:id/apply`：应用新版本。
- [ ] `POST /api/workbenches/:id/rollback`：回滚版本。

### 7.2 会话和工作台关系

- [ ] 会话可以没有工作台，属于普通对话。
- [ ] 会话可以绑定一个工作台。
- [ ] 一个工作台可以有多个会话。
- [ ] 工作台保存时记录来源会话。
- [ ] 工作台重新打开时，把 prompt / memory / layout 注入当前会话。

### 7.3 状态文件

- [ ] 先使用本地 JSON 文件，降低复杂度。
- [ ] 数据落到 `.hermes-home/` 下，和当前 Hermes WebUI 状态隔离。
- [ ] 每次改造保存版本快照。

## 8. Agent 集成 TODO

目标：工作台不是纯前端玩具，必须进入 Agent 的工作上下文。

- [ ] 给每个工作台生成专属 system prompt。
- [ ] 用户进入工作台时，把工作台 prompt 注入 Hermes Agent。
- [ ] 用户在工作台内上传或生成的文件，归属到该工作台。
- [ ] Agent 回复时能知道当前任务面板状态。
- [ ] Agent 可以更新任务面板数据。
- [ ] Agent 可以提出保存或改造工作台建议。

第一版可先用前端状态模拟面板更新，但必须预留 Agent 写入通道。

## 9. 验收演示脚本

第一条完整 demo 用 PPT：

```text
1. 打开产品。
2. 点击新对话。
3. 输入：帮我做一个融资 PPT。
4. AI 追问主题、受众、目标和材料。
5. 用户补充材料。
6. 页面出现 PPT 任务面板：大纲、页面、讲稿。
7. 用户说：第 3 页太散，改得更有冲击力。
8. 面板显示第 3 页被更新。
9. AI 提示：要不要保存为 PPT 工作台？
10. 用户保存。
11. 左侧出现 PPT 工作台。
12. 用户进入 PPT 工作台，说：以后先问我页数和受众。
13. 系统出现改造建议卡。
14. 用户预览、应用。
15. 用户回滚。
```

这条 demo 跑通，才算我们真的有 MVP。
